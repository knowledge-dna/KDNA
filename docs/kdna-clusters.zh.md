# KDNA 集群

KDNA Cluster 是一个可组合的判断系统——多个小型 KDNA 包在明确的角色下协同工作，处理复杂领域任务。

## 为什么不能做一个大 KDNA？

一个覆盖宽泛领域的大 KDNA 有四个问题：

| 问题 | 例子 |
|------|------|
| **太笼统** | 一个包含 20 条原则的"商业增长 KDNA"会变成一堆正确的废话 |
| **上下文浪费** | 每次加载全部判断模式，但当前任务只需要其中 2 个 |
| **框架冲突** | "速度优先"和"信任优先"在同一个包里无声冲突 |
| **无法测试** | 所有东西打包在一起，无法证明哪个具体判断改善了 |

## Package vs Cluster

| | KDNA Asset | KDNA Cluster |
|---|---|---|
| **范围** | 一个判断分岔点 | 一个复杂领域 |
| **文件** | 2-6 个 JSON 文件 | Manifest + 多个 Package |
| **加载** | 加载这个包 | 路由到正确的包 |
| **验证** | kdna dev validate | kdna dev validate |
| **例子** | 价格异议 KDNA | 销售判断集群（5 个包） |

## 集群角色

集群中的每个包有四种角色之一：

| 角色 | 职责 | 规则 |
|------|------|------|
| **Primary（主）** | 当前任务的主判断镜头 | 每次任务只有一个 |
| **Advisor（辅）** | 从另一个角度补充判断 | 每次任务最多 3 个 |
| **Constraint（约束）** | 硬边界，可以覆盖不安全建议 | 检测到风险时加载 |
| **Critic（审查）** | 反向审计主判断的输出 | 审查模式下加载 |

## 组合规则

1. **必须有主判断。** 每次任务有且只有一个 Primary KDNA。
2. **不能平均。** 两个包意见冲突时，暴露冲突——不要折中成"既要又要"的废话。
3. **冲突必须可见。** 告诉用户："包 A 认为这是 X。包 B 认为这是 Y。哪个视角更合适？"
4. **按任务阶段加载。** 诊断→设计→表达→审查，每个阶段加载不同的包。不要一次性全加载。
5. **集群有自己的评测。** 不只测试单个包，还测试集群作为一个系统是否选对了主判断、避免了无关包、识别了冲突、输出了分层判断。

## 集群 Manifest

集群由 `KDNA_Cluster.json` 定义，结构与单包 `kdna.json` 类似，但描述的是包的组合关系。

## 什么时候用集群而不是单包

| 用单包 | 用集群 |
|--------|--------|
| 任务只有一个清晰的判断分岔点 | 任务跨越多个判断维度 |
| 一个专家视角就够 | 不同阶段需要不同视角 |
| 领域窄且边界清晰 | 领域宽且有交互的子领域 |
| 你在创建第一个 KDNA | 你有多个已验证的包 |

## 与现有概念的关系

- **判断模式**：最小单元——信号/误读/框架/边界（定义在 `benchmarks/`）
- **KDNA 包**：包含 2-6 个 JSON 文件的领域包（定义在 `SPEC.md`）
- **KDNA 集群**：有明确角色的可组合包系统（本文档）

这三层互补：模式在包里面，包被组织成集群。
