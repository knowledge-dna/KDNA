# 质量标准

不是每个结构正确的 KDNA 领域都值得发布。本文档定义每个级别质量的真正含义——以及晋级需要什么证据。

## 质量级别

| 级别 | 徽章 | 含义 | 要求证据 |
|------|------|------|---------|
| 0 | `untested` | 结构有效，暂无判断证据 | lint + validate 通过 |
| 1 | `tested` | 人工确认的测试覆盖 | ≥10 个 eval cases |
| 2 | `validated` | 可测量地改善判断 | ≥30 个 eval cases + 公开 benchmark、rubric、原始输出 |
| 3 | `expert_reviewed` | 独立验证 | validated 证据 + 外部领域专家审阅 |
| 4 | `production_ready` | 实战验证 | expert_reviewed 证据 + 30+ 天真实使用、结果数据、无退化 |

## 反空洞门禁

以下模式在 `untested` 结构门禁中就会被拒绝：

| 字段 | 拒绝模式 | 最低标准 |
|------|---------|---------|
| `axioms[].one_sentence` | "信任是重要的。" | 必须描述一个具体判断 |
| `ontology[].essence` | 字典定义 | 必须是该领域中的操作含义 |
| `ontology[].boundary` | 仅否定（"不是不信任"） | 必须指出常与什么混淆 |
| `stances[]` | "要有帮助。" | 必须是能使 Agent 产生偏向的处方 |
| `self_check[]` | "这个回答好吗？" | 必须是领域特定的 yes/no 问题 |
| `misunderstandings[].wrong` | 没人会信的稻草人 | 必须是真实 Agent 可能持有的信念 |

## 验证级要求

要升级到 `validated`，需要一个 benchmark：

- 至少 30 个场景
- 真实模型输出（不是手写期望答案）
- 两种条件：no-KDNA 和 with-KDNA
- 原始输出保留
- 公开报告（含失败案例）
- 可复现的评估脚本

## 专家审阅 Rubric

| 维度 | 问题 | 分数 |
|------|------|:--:|
| 公理准确性 | 公理是否反映专家在该领域的实际判断方式？ | 0-3 |
| 完整性 | 是否遗漏了关键的判断模式？ | 0-3 |
| 操作清晰度 | Agent 能否实际将这些规则应用到文本？ | 0-3 |
| 安全性 | 遵循这些规则可能造成伤害吗？ | 0-3 |
| 总体 | 你会在生产 Agent 中使用这个 KDNA 吗？ | 0-3 |

**最低要求：≥ 10/15**，无任何维度为 0。

## 反模式（永远不要发布这些）

| 反模式 | 示例 | 拒绝原因 |
|--------|------|---------|
| 口号公理 | "沟通是关键。" | 不改变 Agent 行为 |
| 仅否定边界 | "不是坏沟通。" | 不帮助 Agent 区分 |
| 通用自检 | "有帮助吗？" | 任何领域都会问这个 |
| 稻草人误解 | "误解：沟通不重要。" | 没有 Agent 会信 |
| 字典本体 | "信任：对可靠性的坚定信念。" | 从字典复制；不可操作 |
| 凑数量 | 20 条公理，全是一行话 | 数量≠质量 |
| 精挑 benchmark | 只展示 KDNA 赢的场景 | 没有失败案例的 benchmark 是宣传 |
