# 什么时候用 KDNA

不是每个任务都需要 KDNA。当判断比流程更重要时使用它。

## 触发条件

当**同一个表面输入可能意味着多种不同的事情——且判断错误会导致错误行动时**，使用 KDNA。

如果任务有明确流程（步骤、模板、输出格式），用 Skills。
如果任务需要判断"这是什么情况"，用 KDNA。

## 使用 KDNA

| 场景 | 原因 |
|------|------|
| 用户说"你的价格太贵了" | 可能是价值不确定、预算约束或风险恐惧。诊断错误 → 回应错误。 |
| 团队会议记录 | 可能是真正的决策或只是讨论。把讨论误认为决策 → 虚假进展。 |
| 用户请求一个功能 | 可能是真实需求或猜测的方案。做错东西 → 浪费。 |
| 指标下降 30% | 可能是真实信号或正常波动。对噪音行动 → 浪费资源。 |
| 有人要学习计划 | 可能需要结构或需要反馈。没有反馈的计划 → 注定失败。 |

## 不用 KDNA

| 场景 | 原因 | 用什么 |
|------|------|--------|
| 把这篇文章格式化成表格 | 纯格式化任务 | Skills |
| 法国的首都是什么？ | 事实查询 | 搜索 / RAG |
| 用更友好的语气写 | 风格调整 | Prompt / Skills |
| Markdown 转 HTML | 机械转换 | Skills / 工具 |
| 根据这些数据生成周报 | 模板输出 | Skills |

## 边界规则

不确定时问自己：**"一个专家和一个新手会对这个输入产生不同的解读吗？这种差异会改变采取的行动吗？"**

如果是 → KDNA。如果否 → Skills、RAG 或 Prompt。
